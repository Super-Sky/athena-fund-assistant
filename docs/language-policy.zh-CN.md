# 文档与代码注释双语规范

## 目标

本仓库面向中文产品讨论和英文工程协作。长期文档必须支持中文和英文两个版本，避免只有单语文档导致产品、合规、工程实现之间语义漂移。

## 文档规则

稳定文档采用双文件结构：

- `topic.zh-CN.md`
- `topic.en-US.md`

例如：

- `product-boundary.zh-CN.md`
- `product-boundary.en-US.md`

要求：

- 两个版本必须表达同一组事实、边界和验收标准。
- 可以根据语言习惯调整表达顺序，但不能改变结论。
- 修改任一版本时，必须检查另一个版本是否需要同步。
- `docs/README.md` 应同时链接两个语言版本。
- 历史单语文档可以作为草稿保留，但完成交付前应收敛成双语版本。

## 代码注释规则

核心类型、导出函数、复杂业务规则和容易误用的逻辑必须写双语注释。

格式要求：

```ts
// DecisionOption describes one actionable fund decision branch.
// DecisionOption 描述一个可执行的基金决策分支。
export interface DecisionOption {}
```

要求：

- 英文在前，中文在后。
- 两种语言分成两行，不写在同一行。
- 不为显而易见的代码补空洞注释。
- 注释应说明“为什么存在、边界是什么、容易影响哪里”，而不是重复字段名。

## 命名规则

- 代码文件名、包名、配置名默认使用英文。
- 用户可见文案可以由产品层提供中英文资源。
- 非工具入口文档优先使用 `*.zh-CN.md` 和 `*.en-US.md` 后缀。

## 交付门禁

涉及以下内容时，交付前必须检查双语同步：

- 产品边界
- 金融治理规则
- 数据源与授权假设
- Athena 对接契约
- 领域模型
- MVP 验收标准
- 用户可见风险提示
